9.1 The VIM 2 Card: Six LEDs That Tell a Story

The VIM 2 (Versatile Interface Module, version 2) card is a powerful communication interface that bridges DeltaV controllers with external devices over VIMNet Ethernet networks. Unlike most I/O cards that have a simple Power/Error/Channel pattern, the VIM 2 has six LEDs that together paint a complete picture of the card's health, redundancy status, and communication state.

From top to bottom, the six LEDs are:

LEDColorNormal StateMeaning
PowerGreenOnCard is powered. If Off, provide power to the card.
ErrorRedOffWhen On, the VIM 2 is in error; all other LED states are not significant.
ActiveGreenOn (if active in redundant pair)Card is commissioned and operating as the active card in a redundant pair.
StandbyGreenOff in simplex mode; On if standby in redundant pairCard is commissioned and operating as the standby card in a redundant pair.
NetworkAmberOn (solid) or BlinkingOn = valid VIMNet network communications. Blinking = no valid IP address (decommissioned).
Ctlr CommAmberOnValid LocalBus (controller) communications.
Key Concept — The VIM 2 Redundancy Rule: If the Active LED is On and the Standby LED is Off, the card's redundancy role is Active. If the Active LED is Off and the Standby LED is On, the role is Standby. Both LEDs On simultaneously is not a normal state. In simplex mode, the Standby LED is always Off.

⚠️ Common Misdiagnosis: A VIM 2 with the Network LED blinking and Ctlr Comm Off might be mistaken for a completely failed card. In reality, the card itself may be fine — the problem is typically that the controller has not been downloaded, or the external device configuration has not been uploaded from VIMNet Explorer. Always check the software configuration before replacing hardware.

9.3 Smart Switch LEDs: MD20 and MD30

The DeltaV MD20 and MD30 Smart Switches are the backbone of the Control Network. They have two categories of LEDs: device status LEDs that describe the health of the base switch, and display status LEDs whose meaning changes depending on which item you have selected using the Select button on the base switch.

Every DeltaV media module has one LED per port. The meaning of these port LEDs depends on which item is currently selected on the base switch's display. Press the Select button to cycle through items.

Key Concept — The Select Button: The MD20/MD30 base switch has a Select button that cycles through display modes (L/D, FDX, 1000, AN, TP, PoE). The port LEDs on the media modules change meaning with each mode. If you are reading port LEDs and they seem inconsistent, check which mode is currently selected on the base switch. The most common troubleshooting mode is L/D (Link/Data status).

9.6 DeltaV Diagnostics: The Software Troubleshooting Layer

After you have completed the physical checkout (Lesson 7) and verified LED indicators (Lessons 8 and 9), you move to the software layer. DeltaV provides three primary tools for software-level verification:

9.6.1 DeltaV Explorer Status Indicators

When you open DeltaV Explorer and look at the Control Network tree, you will see status indicators next to each node. These indicators tell you whether the database matches the physical hardware:

IndicatorMeaningCorrective Action
Installing iconThe downloading workstation does not have all the information about this nodeClick the node, right-click, select "Install Setup Data"
Not Communicating iconThe node is not communicating (bad connection, not powered up, or decommissioned)Check LEDs on the node; verify cables are connected to correct ports; check power
Integrity Problem iconThe node is communicating but has an integrity problem (I/O mismatch, missing network connections, or Event Chronicle issues)Check LEDs; use DeltaV Diagnostics "Check Node Integrity"; compare I/O configuration vs. physical I/O
No Configuration iconThe node has never been downloaded or lost its configurationDownload the node
⚠️ Common Mistake: Do not change your configuration when you see an integrity problem indicator. The problem is almost always a hardware mismatch — the I/O cards physically installed do not match what is configured in the database. Use the Auto-sense I/O Cards command (right-click on the I/O node → Auto-sense I/O Cards) to compare what the controller sees versus what the database expects.

9.7 Getting Help: Resources and Support

When the self-service troubleshooting tools cannot resolve your issue, Emerson provides several support channels:

  • Online Help — Available in every DeltaV application via the Help menu. Provides procedural help and popup help for dialog boxes.
  • Books Online — The comprehensive DeltaV documentation library, accessible from Start → DeltaV Help → Books Online. The System Administration and Maintenance manual is particularly useful for hardware troubleshooting.
  • Technical Service — Help desk support, remote diagnosis, 24-hour emergency support, and software update service. Contact via the Emerson Automation Solutions website.

When contacting technical support, have the following information ready:

  • System Identification Number
  • Software Version Number
  • Description of the problem
💡 Tip: If possible, be at your DeltaV system when calling technical support. This enables the representative to step you through diagnostic procedures in real time, which is significantly faster than describing the problem over the phone.

9.9 Complete Troubleshooting Workflow Summary

This three-lesson module (Lessons 7–9) has covered the complete hardware checkout and troubleshooting process. Here is the end-to-end workflow:

  1. Cable verification — Test all network cables end-to-end with the Microtest PentaScanner (Lesson 7, Section 7.2).
  2. Power verification — Measure voltages at PSU, last carrier, and field power terminals; test redundant supplies independently (Lesson 7, Section 7.3).
  3. LED inspection — Read Power, Error, and channel LEDs on every device and card (Lessons 8 and 9, Sections 8.1–9.5).
  4. Field wiring test — Use test probes at I/O terminal block access points (Lesson 7, Section 7.5).
  5. Software startup — Launch DeltaV Explorer and verify node communication status (Lesson 9, Section 9.6).
  6. Database verification — Compare configured I/O vs. physical I/O using Auto-sense I/O Cards (Lesson 9, Section 9.6.1).
  7. Escalation — Contact Emerson Technical Service with System ID, software version, and problem description (Lesson 9, Section 9.7).

Quiz

1. What are the six LEDs on the VIM 2 card, listed from top to bottom?

Click to reveal answer

(1) Power (green), (2) Error (red), (3) Active (green), (4) Standby (green), (5) Network (amber), (6) Ctlr Comm (amber).

2. What does it mean when the VIM 2 Network LED is blinking but all other LEDs are in their normal states (Power On, Error Off, Active/Standby in correct state)?

Click to reveal answer

A blinking Network LED indicates the VIM 2 does not have a valid IP address — it is likely decommissioned. Use VIMNet Explorer to configure a placeholder with the correct IP address, then commission the VIM 2.

3. What distinctive LED pattern indicates that no terminal block is installed (or the wrong type is installed) on an I.S. DI 16-Channel card?

Click to reveal answer

A "running light" pattern where the yellow channel LEDs light up one by one from Channel 1 to Channel 16, with the Red Error LED simultaneously On.

4. On the MD20/MD30 Smart Switch, what do the port LEDs mean when the display is set to "L/D" mode?

Click to reveal answer

In L/D (Link/Data) mode: green = valid connection, not glowing = no valid connection, flashing green once per period = port in standby, flashing green three times per period = port locked by lockdown application, flashing yellow = port receiving data.

5. What is the difference between the Power LED on an RM100/FP20 showing green vs. yellow?

Click to reveal answer

Green means both power supplies are on and healthy. Yellow means only one supply voltage is on — the switch is running on a single supply with no redundancy margin. The failed supply should be investigated immediately.

6. When you see a "Not Communicating" indicator next to a controller in DeltaV Explorer, what are the first three things to check?

Click to reveal answer

(1) Check the LEDs on the physical controller. (2) Check the Control Network cables — make sure they are connected to the correct ports on the switch and controller. (3) Verify that primary cables connect to primary ports and secondary cables to secondary ports (they may be crossed).

7. What does the Auto-sense I/O Cards command do, and when should you use it?

Click to reveal answer

Auto-sense I/O Cards compares the I/O cards physically installed in the controller (auto-sensed type) against what is configured in the database. Use it when you see an integrity problem indicator to identify mismatches between physical and configured I/O.

8. What three pieces of information should you have ready when contacting Emerson Technical Service?

Click to reveal answer

(1) System Identification Number, (2) Software Version Number, and (3) Description of the problem.

9. How can you identify which physical controller corresponds to a node in DeltaV Explorer?

Click to reveal answer

Right-click the controller in DeltaV Explorer → Node Properties → Identify Controller. This causes the LEDs on the physical controller to flash, confirming you have found the right device.

10. Complete the troubleshooting workflow: After verifying cables and power (physical layer) and inspecting LEDs (device layer), what is the next step before calling technical support?

Click to reveal answer

Start the DeltaV software and verify the node communication status in DeltaV Explorer. Use the status indicators to identify whether nodes are communicating, whether integrity problems exist, and whether configuration is complete. Use Auto-sense I/O Cards to compare physical vs. configured I/O. Only after exhausting software-level diagnostics should you escalate to technical support.
